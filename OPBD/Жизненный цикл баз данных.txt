3.1.1 Планирование разработки базы данных
	Содержание данного этапа - разработка стратигического плана, в процессе которой осуществляется предварительное планирование процессе системы управления базами данных.
	Планирование состоит в определении трёх основных основных компонентов: объема работ, ресурсов и стоимости проекта.
	Первая часть - проверка технологий осуществимости. Она состоит в вяснениии запроса.
	Вторая часть - проверка операционной осуществимости - выяснение наличия экспертов и персонала, необхадимых для работы БД.
	Третья часть - проверка экономической целесообразности осуществления проекта. При исследовании этой проблемы важно дать оценка ряду факторов:
		*целесообразность совместного использования базы данных разными отделами.
		*Величина риска, связанного с реализацией системы БД.
		*Ожидаемая выгода.
		*Время окупаемости
		*Влияние системы управления БД на релизацию долговременных планов организации.


3.1.2 Определение требований к системе.
	Определение включает выбор целей БД, выяснение информационных потребностей различных отделов и руководителей фирмыи требований к оборудованию и ПО.


3.1.3 Сбор и анализ требований к системе
	Необхадимо создать для себя модель движени важныз материальных объектов и уяснить процесс документооборота. По каждому документу нужно определить переодичность использования, определить данные, необхадимые для выполнения выделенных функций


3.1.4 Проектирование базы данных
	Полный цикл разработки базы данных включает концептуальное, логическое и физическое её проектирование.
	Концептуальное проектирование БД.
	Первая фаза заключается в создании для анализируемой части предприятия концептуальной модели данных. ПРоектирование слажных БД осуществляется с использованием нисходящего подхода.
	Этот подход начинается с разработки моделей данных, которые содержат несколько высокоуровневый сущностей и связей, затем работа продолжается в виде серии нисходящих уточнений низкоуровневых сущностей, связей и относящихся к ним атрибутов.
	Нисходящий подход демонстируется в концепции модели "Сущность - связь", оно относиться в сематическим моделям. Это связанное со смысловым содержанием данных, независимо от их представления в ЭВМ.

	Общие концептуальные модели данных:
		*Выделение локальных представлений. существующих обычно относительно независимым данным.
		*Формулирование сущностей, описывающий локальную предметную область проектируемой БД.
		*Выделение ключевых атрибутов.
		*Спецификация связей между сущностями. Удаление избыточных связей.
		*Анализ и добавление неключевых требований.
		*Объединение локальных представлений
	Логическое проектирование БД.
	Логическая модель, отражающая особенности представления о функционировании предприятия одновременно многих типов пользователей, наз-ся глобальной логической моделью данных.
	Процесс проектирования БД должен опираться на определённую моделью модель данных, которая определяется типом предлагаемой для реализации информационной системы СУБД.
	Концептуальное и логическое проектирование - это итеративные процессы, которые включают в себя ряд уточнений, продолжающихся до тех пор пока не будет получен наиболеее соответствующий структуре предприятия продукт.
	
	Физическое проектированеи БД.
	Целью проектирования на данном этпапе является создание описания СУБД ориентрованной модели БД.
	Действия, выполняемые на этом этапе, слишком специфичны для раздичных моделей данных, поэтому сложно обобщить. Остановимся на реляционной модели данных. В этом случае под физ. проектированием подразумевается:
		*Создание описания выбора набора реляционных таблиц и ограничений для них на основе информации.
		*Определение конкретных структур хранения данных и методов доступа к ним.
		*Разработка средств защиты создаваемой системы.
3.1.5 Разработка приложений.
	Главные составляющие данного процесса - это проектирование транзакций и пользовательского интерфейса.

	Проектирование транзакций
	Транзакция может состоять из нескольких операций, однако с точки зрения пользователя эти операции представляют собой единое целое.
	Проектирование транзакций заключается в определении:
		*Данных, которые используются транзакцией 
		*функциональных характеристик транзакции
		*выходных данных, формируемых транзакцией
		*степени важности и интенсивности использования транзакции.
	Проектирование пользовательского интерфейса
		*Содержательное название
		*ясные и понятные инструкции
		*логически обоснованные группировки и последовательностти полей
		*визуально првилекательный вид окна формы или поля отчёта
		*легко узнаваемые названия полей
		*согласованную терминологию и сокращения; 
		*согласованное использование цветов; 
		*визуальное выделение пространства и границ полей ввода данных; 
		*удобные средства перемещения курсора; 
		*Средства испрвления отдельных ошибочных символов и целых полей
		*Средства вывода сообщений об ошибках при выводе 
		*особое выделение необязательных для ввода полей
		*сред
